# Evolve Skill — 半自动技能进化

> 升级版：新增 **自动触发**（由 skill-health-monitor 低分告警拉起）和 **三档安全策略**（微小改进自动写回、中等改进展示确认、无改进仅展示）。

---

## 触发方式

### A — 自动触发（由健康度监测拉起）
当 `skill-health-monitor` 检测到某个技能健康度下降时，会自动标记建议进化。我将在以下时机自动启动进化：
- 你说了"好"/"行"/"开始进化"等确认
- 健康度报告展示后你无异议

被触发的进化使用**简化流程**（跳过 Phase 1 分析，直接进入变异），因为健康度数据已经指明了问题方向。

### B — 手动调用
```
/skill evolve-skill <目标技能名>
```
不指定技能名则列出所有可选技能。

---

## 进化流水线

### Phase 0: 健康度检查（自动触发时）

如果由 skill-health-monitor 触发，先读取健康度记录：
- 是哪个维度出了问题？（完成度/流畅度/用户反馈）
- 进化方向优先聚焦该维度
- 如果无健康度数据（手动调用），跳过此步

### Phase 1: 读取 & 分析

1. 用 `read_file` 读取 `.reasonix/skills/<name>.md`
2. 提取 YAML 元数据
3. 分析评估四维度：清晰度、完整性、可执行性、效率
4. 输出分析摘要

### Phase 2: 变异（生成改进版本）

生成 **2~3 个变异版本**：

| 变异 | 侧重 |
|---|---|
| v1 — 简洁优化 | 精简冗余，降低 token 消耗 |
| v2 — 功能增强 | 补充缺失步骤和边界情况 |
| v3 — 结构重组（可选） | 重新组织逻辑层次 |

约束：
- ✅ 保持 YAML frontmatter 完整
- ✅ 不超过原文件 120% 大小
- ✅ 如健康度指明问题方向，优先修复该维度

### Phase 3: 评估（LLM-as-Judge）

构造 3 个测试用例，对原版 + 各变异评分（每项 1-10）：

- **清晰度** | **完整性** | **可执行性** | **效率**

汇总得综合分。

### Phase 4: 择优

确定优胜版本。计算相对于原版的：
- 总分差值 Δ
- 内容变动比例

### Phase 5: 三档写回策略 🚦

根据 Δ 和变动比例决定行为：

| 档位 | 条件 | 行为 |
|---|---|---|
| 🟢 **自动写回** | Δ ≥ +2 分 **且** 改动 ≤ 20% | 自动执行：备份旧版 → `edit_file` 写回 → `remember` 记录进化历史 → 回复末尾一行摘要 |
| 🟡 **展示确认** | +1 ≤ Δ < +2，或改动 20%~50% | 展示 DIFF，问你"要写回吗？" |
| 🔴 **仅展示** | Δ < +1，或改动 > 50% | 展示结果，不做任何写回 |

**自动写回的安全保障：**
- 写回前自动用 `remember` 备份原版（name: `evolve-backup:<技能名>:<时间戳>`）
- 随时可说"回滚 <技能名>"恢复
- 改动 > 50% 的自动降级为 🟡 展示确认

## 进化历史追踪

每次进化后（无论自动还是确认后），记录到项目记忆中：

```
remember({
  type: "feedback",
  scope: "project",
  name: "evolve-history:<技能名>",
  description: "<技能名> 进化历史 (第 N 次)",
  content: JSON 记录：时间、原版评分、新评分、Δ、是否被接受
})
```

积累的数据供 `meta-evolve` 分析用：哪种变异策略最有效？评分权重是否合理？

---

## 与 skill-health-monitor 的关系

```
skill-health-monitor 检测到低分
    ↓
标记"建议进化" (alerted = true, 记录问题维度)
    ↓
我自动检查告警 → 符合条件时提示你
    ↓
你确认 → evolve-skill 启动 (Phase 0 读取健康度数据，聚焦问题维度)
    ↓
进化完成 → 更新 skill-health-monitor 记录 (清除告警，重置评分基线)
```
