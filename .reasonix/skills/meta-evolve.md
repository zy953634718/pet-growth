# Meta Evolve — 元进化闭环

> 这是 Hermes 自进化框架的最后一层：**进化系统自身的进化**。
> 通过分析 evolve-skill 的历史表现数据，持续优化进化策略本身。

## 工作原理

```
evolve-skill 每次运行 → 记录进化历史
         ↓
数据积累 ≥ 5 条 → 触发元进化分析
         ↓
分析：哪种变异策略最有效？评分权重合理吗？
         ↓
输出优化建议 → 自动调整 evolve-skill 的参数
         ↓
(循环) → 下一次进化使用优化后的参数
```

## 触发方式

### 自动触发
当以下条件满足时，我会在 auto-reflect 的静默检查中自动运行 meta-evolve：
- 累计进化历史 ≥ 5 条
- 且距上次元进化已 ≥ 3 次进化
- 结果以一行摘要附在回复末尾

### 手动调用
```
/skill meta-evolve
```
输出完整分析报告。

## 分析流程

### Phase 1: 读取进化历史

用 `recall_memory` 读取所有 `evolve-history:*` 条目，提取：

| 字段 | 说明 |
|---|---|
| 技能名 | 被进化的目标 |
| 时间 | 进化发生的时机 |
| 原版评分 | 进化前的综合分 |
| 新版评分 | 进化后的综合分 |
| Δ | 评分变化 |
| 接受状态 | 自动写回 / 确认写回 / 拒绝 |
| 变异方向 | 用了哪个变异策略（v1 简洁/v2 增强/v3 重组） |

### Phase 2: 关键指标计算

| 指标 | 公式 | 意义 |
|---|---|---|
| 接受率 | 已接受 / 总数 | 进化质量的整体指标 |
| 平均 Δ | Σ(Δ) / 总数 | 每次进化平均带来多少改进 |
| 策略胜率 | 各策略被选中的次数 / 涉及该策略的总次数 | 哪种变异方向最有效 |
| 退化率 | Δ ≤ 0 的次数 / 总数 | 进化变差的概率 |

### Phase 3: 优化建议生成

基于指标，生成针对 evolve-skill 的优化建议：

**可优化参数：**
- 各评分维度的权重（目前四维均等，如"效率"不重要可降低）
- 变异策略的优先级（如"简洁优化"总是胜出可优先使用）
- 自动写回的 Δ 阈值（如接受率很高可降低到 +1.5）
- 测试用例数量（如 3 个不够稳定可增加到 5 个）

**输出格式：**
```
━━━ 元进化报告 ━━━

📊 基础数据
  进化总次数: 7 | 接受率: 71% | 平均 Δ: +1.8

📈 策略效果排名
  v1 简洁优化  🥇 胜率 80%  avg Δ +2.1
  v2 功能增强  🥈 胜率 60%  avg Δ +1.5
  v3 结构重组  🥉 胜率 33%  avg Δ +0.8

⚙️ 优化建议
  → 建议将"效率"维度权重从 1.0 降至 0.7
  → 建议将自动写回阈值从 +2.0 降至 +1.5（接受率良好）
  → 建议默认优先使用 v1 变异策略
```

### Phase 4: 应用优化（自动 / 确认）

**自动应用的条件（全部满足）：**
1. 数据量 ≥ 5 条
2. 建议仅涉及参数调整（不改变 evolve-skill 的结构）
3. 改动内容 < 20%

**否则展示确认。**

应用方式：用 `edit_file` 更新 `.reasonix/skills/evolve-skill.md` 中的相关参数。

## 元进化自身的追踪

每次元进化后，记录到项目记忆：
```
remember({
  type: "feedback",
  scope: "project",
  name: "meta-evolve-history",
  description: "元进化历史 (第 N 次)",
  content: JSON：时间、分析的进化数、报告摘要、是否应用了优化
})
```

这构成了第二层闭环 —— 元进化自身的历史也被追踪，使得"元进化的进化"成为可能。

---

## 安全约束

- **永不修改非参数内容** — 只调整数字参数（权重、阈值、数量），不修改 evolve-skill 的结构和流程
- **大幅调整展示确认** — 如果建议涉及删除或重构某个阶段，需你确认
- **每次优化前自动备份** — 用 `remember` 记录旧参数
